# Rumour App 

Rumour is a highly modular, natively robust anonymous chat application built using **Flutter**,  powered completely by **Firebase Cloud Firestore**, and strictly governed by **Clean Architecture** patterns utilizing **BLoC** State Management. 

The application flow strictly enforces total device-bound anonymity routing exclusively using dynamically generated fake randomuser personas locking directly to physical hardware without painful standard email/password authentication barriers.

## đź“± Codebase Architecture (Strict Clean Architecture)
The source repository logic explicitly separates isolated responsibilities mapping perfectly into the native Flutter `lib/features/` framework utilizing **get_it** as the unified Service Locator pattern:

### 1. Presentation Layer (`/presentation`)
- **UI Screens/Components:** Completely decoupled, stateless UI matrix explicitly replicating strict minimal flat glassmorphic Figma constraints matching the dark aesthetic. 
- **BLoC Processors:** Highly reliable stream-bound architectures (`IdentityBloc`, `RoomBloc`, `ChatBloc`) managing the cascade.

### 2. Domain Layer (`/domain`)
- **Entities:** Absolute raw payload containers scaling easily (`UserIdentity`, `Room`, `Message`).
- **UseCases:** Independent execution modules handling single-responsibility actions exclusively (`SaveIdentity`, `GetMessages`).
- **Repositories Maps:** Rigid logical interfaces forcing the presentation layers completely blind to data implementations.

### 3. Data Layer (`/data`)
- **Implementations:** Clean binding interfaces parsing REST JSON payloads seamlessly into dart Models.
- **DataSources:** Hybrid layers safely pushing `FirebaseFirestore` payloads seamlessly aside `SharedPreferences` interceptors (caching the anonymised Identity keys cleanly into device memory bounding isolation automatically per phone!).

## đź—„ Firebase Cloud Firestore Structure
To properly orchestrate the real-time websocket pipelines into Firebase effortlessly, the Database natively tracks state using the following explicitly modeled mapping structure:

### Primary Collection: `rooms`
- `Document ID:` User supplied String Code (e.g. "994513")
- `fields:`
  - `createdAt` [Timestamp] : Core document timestamp enforcing room life-cycles.

### Nested Subcollection: `rooms/{roomId}/messages`
- `Document ID:` FireServer Autogenerated Hash Key String
- `fields:`
  - `text` [String]: Raw message character payloads.
  - `senderId` [String]: SharedPreferences locally cached literal UID bound tightly to the hardware.
  - `senderName` [String]: A randomly generated API username generated during the very first boot instance.
  - `senderAvatar` [String]: Web image pointer pointing cleanly back to generated RandomUser imagery.
  - `createdAt` [Timestamp]: FireServer execution timestamp forcing strict real-time chronological ordering in chat cascades.
